主題 2 錯題（10 題）

錯題 1：Re-render 的因果關係

我曾答：「UI 更新時就會 re-render」

校正：因果反了。Re-render 是 function 重新執行，產生新 Virtual DOM，比對後才決定要不要更新真實 DOM。Re-render ≠ UI 更新。

💡 面試陷阱：被問到「re-render 會發生什麼？」要從「JS 執行 → Virtual DOM → reconciliation → DOM 更新」完整講出來。

錯題 2：useState setter 一定會 re-render？

我曾答：「會」

校正：不會。React 用 Object.is 比較新舊值，相同就 bail out。但物件 / 陣列要小心：setUser({ name: 'Bo' }) 是新物件 reference，會 re-render。

錯題 3：父 re-render，子不 re-render 的條件

我曾答：「父的 state 更新，子的 state 跟原本相同時」

校正：子的 state 完全無關。需要兩個條件同時成立：
① 子元件被 React.memo 包起來
② 傳給子的 props 在淺比較下沒改變

錯題 4：React.memo 的作用

我曾答：「讓沒有用到 props 的 UI 不會跟著 re-render」

校正：「讓元件在 props 沒有改變時跳過 re-render」。差別：元件可能有 props，只是這次沒變。

錯題 5：count 改變時 React.memo 包住的 Dashboard 會不會 re-render？

我曾答：「會，因為 memo 包住的 props 是 config 而不是 count」

校正：理由錯。memo 不會看 count（那不是 Dashboard 的 prop）。真正原因：父 re-render 時 const config = { ... } 重新建立新物件，config 的 reference 變了，淺比較失敗。修法：用 useMemo 穩定 config。

錯題 6：分不清 useCallback / useMemo

我曾答：把 const title = useMemo(() => 'User Dashboard', [...]) 寫成 useCallback——被「箭頭函式」誤導。

校正判斷：看等號左邊變數最後拿到什麼。函式 → useCallback；值（物件、陣列、字串、計算結果）→ useMemo。
第二個錯：簡單字串拼接根本不該用 useMemo（primitive 不需要穩定 reference）。

錯題 7：Cleanup 執行的時機

我曾答：「依賴的值有變動時」

校正：漏掉一個重要時機。完整兩個時機：
① 下次 effect 即將執行前（依賴變動）
② 元件 unmount 時 ← 我漏了
同一段 cleanup 同時解決 race condition（時機 1）和 memory leak（時機 2）。

錯題 8：Custom hook 的好處

我曾答：「好維護」（太簡略）

校正：完整四個好處——邏輯重用、關注點分離、可測試性、集中升級邏輯入口（最重要的價值）。

錯題 9：Debounce / Throttle 判斷

我曾誤判 2 題：地圖座標、字數顯示，都答 debounce。

校正關鍵字：「動態 / 即時 / 過程中持續顯示」 = Throttle；「停下來才 / 等使用者完成」 = Debounce。
進階：字數顯示連 throttle 都不需要——text.length 是免費操作，硬包反而是反優化。

錯題 10：Virtualization 的判斷

我曾答：「450 萬筆紀錄不用 virtualize，因為數量不影響效能」

校正：450 萬筆直接渲染瀏覽器會 OOM。閾值：<50 直接渲染、50-200 加 memo、200-2000 該 virtualize、2000-10000 必須 virtualize + 分頁、>10000 必須 virtualize + 必須分頁。
更深的層次：450 萬筆根本不該渲染列表，應該用聚合儀表板（圖表 + 統計）呈現。最厲害的設計是讓使用者根本不需要面對大資料。

主題 1 錯題（2 題）

錯題 11：Auth Token 是哪種 state？

我曾答：「Server State，需要從後端發出」

校正：是 Global Client State。Token 雖然來自 server，但拿到後存在前端、不需要反覆 fetch、主控權在前端。
核心觀念：「資料來源」是粗糙標準，「同步模式」才是精確判斷。

錯題 12：為什麼選 Pinia？

我曾答：「因為這是 Vue 的預設」

面試忌諱：傳達「沒比較過、沒思考過」的訊號。
校正答法：「Pinia 是 Vue 3 官方推薦，TypeScript 友好、API 簡單、跟 Composition API 整合度高。對 Myverse 這種規模 boilerplate 比 Redux 少。但 server state 我會考慮 Vue Query——Pinia 自己實作 cache、retry 會增加維護成本。」

主題 3 錯題（3 題）

錯題 13：「每次編輯都會出現」AI 提示

我曾答：每次編輯都跳

校正：會變成狼來了——使用者開始無視所有提示。需要：降低觸發頻率、同時最多顯示 N 條、使用者忽略後變淡。

錯題 14：AI 需要看的資料只列 3 層

我曾答：UI State、Analytics、Audience

校正：漏了 Creator Identity（沒它 AI 給不出針對性建議）和 User Intent Signals（隱性訊號：剛刪了什麼、在哪卡片停很久、對話歷史）。完整 5 層。

錯題 15：AI 延遲只想到 1 個解法

我曾答：「輸入中圖示」

校正：那只是 loading indicator 一層。完整四層：
① Loading indicator
② Streaming（核心技巧）
③ Process transparency（進階）
④ Confidence indicator（最常被忽略）

1State 三層架構（最核心）

核心觀念：Client State vs Server State

⚠ 我曾完全沒聽過這個區分

這是現代前端的核心觀念。不懂它，state 工具會一直選錯；懂了，很多選擇變得理所當然。

特性Client StateServer State
資料來源由前端產生來自伺服器
真實性前端就是真相前端只是「快取」，真相在 DB
同步性同步、立即可用非同步、要 fetch
失敗模式幾乎不會失敗網路錯誤、權限錯誤、過時
多人共用沒這問題別人也會改它
例子theme、表單輸入、彈窗開關使用者資料、商品列表

精確的判斷流程

Q1：前端會反覆重新拉嗎？ ├─ 會（要保持最新）→ Server State └─ 不會（拿到後一直用到失效）→ 進 Q2 Q2：主控權在誰？ ├─ Server（會被別人改）→ Server State └─ 前端（前端是唯一修改者）→ Client State Q3（已確定 Client State）：誰需要它？ ├─ 跨元件 → Global Client State └─ 單一元件 → Local State
💡 核心觀念

「資料來源」是粗糙訊號，「同步模式」才是真正的判斷標準。

三層 State 架構（面試金句）

┌─ Server State ──────────────────────────┐ │ 工具：React Query / Vue Query / SWR │ │ 資料：使用者、商品、訊息、API 來的東西 │ └─────────────────────────────────────────┘ ┌─ Global Client State ───────────────────┐ │ 工具：Pinia / Zustand / Redux / Context │ │ 資料：theme、auth token、UI 偏好 │ └─────────────────────────────────────────┘ ┌─ Local State ───────────────────────────┐ │ 工具：useState / ref │ │ 資料：表單輸入、彈窗開關、hover │ └─────────────────────────────────────────┘

最常見反模式：用 Redux / Pinia 管 server state——導致大量 boilerplate（loading、error、cache 都要手寫），且容易跟伺服器資料不同步。

Context API 的效能陷阱

Context 是「整包訂閱」——任何訂閱者在 value 變動時都會 re-render，不支援 selector 精細訂閱。

// Zustand 支援 selector
const color = useThemeStore(state => state.color)
// 只在 color 變動時 re-render，size 變動不影響

Context 適合放：「幾乎不變動」的東西（theme、locale）

頻繁變動的全域 state：用 Zustand 或 Redux

Context / Zustand / Redux 選擇

你的 client state 多複雜？ 簡單（少數靜態資料：theme、locale）→ Context API 中等（多模組、需精細效能、團隊小）→ Zustand 複雜（大團隊、複雜變更邏輯、需 middleware）→ Redux Toolkit

但前置問題永遠是：「這個 state 是 client 還是 server？」如果是 server，三個都不該用，應該用 React Query。

Myverse 的 state 重新分類

分層Myverse 的資料
Server State
(Vue Query)		profile、links、stats、訂閱方案
Global Client State
(Pinia)		auth token、登入使用者基本資訊（user.id）、編輯/預覽模式切換、全站 toast
Local State
(ref)		表單輸入暫存、彈窗開關、hover
⚠ 微妙區分

user.id 是 Client State（拿到後不變），但 profile.display_name 是 Server State（隨時會被改）。

面試版回答

現代前端 state management 的核心是「分層」，根據 state 的本質選工具：

Server State 是「前端只是快取，真相在後端」——profile、商品、訊息。需要 cache、retry、去重、樂觀更新。我會用 React Query / Vue Query。

Global Client State 是「前端是唯一主控者，跨元件共享」——auth token、theme、全站 toast。簡單用 Context、中等用 Zustand、複雜用 Redux Toolkit。

Local State 是「只關心單一元件」——表單輸入、彈窗開關。用 useState。

最常見反模式是用 Redux 管 server state，會導致大量 boilerplate。Context 也有陷阱：整包訂閱，所有訂閱者在 value 變動時都 re-render，所以我用它放幾乎不變的東西，頻繁變動的全域 state 用 Zustand。

判斷一個 state 屬於哪一層，看的是同步模式而不是資料來源——例如 auth token 雖來自 server，但拿到後幾乎不變、不需要 cache 機制，是 Global Client State。

2Multi-view 資料同步

題目原文："How you approach data synchronization across multiple views when the underlying data changes frequently"

場景：Myverse 編輯頁的三個 view

┌─────────────────────────────────────────────────┐ │ [頂部導覽列] 🙂 Bo Kai │ ← View 3 ├─────────────────────────────────────────────────┤ │ [編輯區] │ [手機預覽] │ │ Display Name: ___ │ ┌──────────┐ │ │ Bio: __________ │ │ 🙂 │ │ ← View 2 │ Avatar: [上傳] │ │ Bo Kai │ │ │ ↑ View 1 │ └──────────┘ │ └─────────────────────────────────────────────────┘

核心原則 1：Single Source of Truth

各 View 各自 fetch 會有三個問題：

  1. 浪費網路（同樣資料 fetch 三次）
  2. 資料可能不一致（fetch 時間點不同）
  3. 更新時要通知所有 View（指數級複雜度）

正解：所有 View 訂閱同一份資料。

核心原則 2：Publish-Subscribe

  • State 是 publisher
  • 所有 View 是 subscriber
  • 變動時 library 自動通知所有 subscriber

Vue 響應式：自動追蹤（用到就追蹤）
React 響應式：手動訂閱（用 hook 選擇要訂閱什麼，更精細）

三種更新策略

1. Pessimistic Update（悲觀更新）

按儲存 → 顯示 loading → 等 server → 成功才更新 UI
  • ✅ 不會 UI / server 不一致
  • ❌ 體驗慢
  • 適合：刪除、付款、權限變更

2. Optimistic Update（樂觀更新）

按儲存 → 立刻更新 UI → 背景呼叫 server → 失敗 rollback
  • ✅ 反應極快
  • ❌ rollback 邏輯複雜，失敗時 UI 跳回去使用者會困惑
  • 適合：按讚、加收藏、編輯文字

3. Pending State（待確認）

按儲存 → 顯示新值 + 標示「儲存中」icon → 等 server → 成功移除標示，失敗 rollback
  • ✅ 兼顧速度感和狀態誠實
  • ❌ UI 設計成本高
  • 適合：即時通訊、協作工具

Cache Invalidation（核心難題）

const mutation = useMutation({
  mutationFn: updateProfile,
  onSuccess: () => {
    queryClient.invalidateQueries({ queryKey: ['profile'] })
    // 訂閱這個 query 的 view 自動重新 fetch
  }
})

進階：cache 之間的依賴關係

// Myverse：改 avatar 後
onSuccess: () => {
  queryClient.invalidateQueries({ queryKey: ['profile'] })  // 自己
  queryClient.invalidateQueries({ queryKey: ['links'] })    // 連結列表也顯示縮圖
}

忘了 invalidate ['links'] → 「個人區頭像變了，但連結列表小頭像還是舊的」← 典型 multi-view 不同步 bug。

Normalized Cache（最高難度）

問題：每次 invalidate 都要重新 fetch，浪費頻寬。

傳統巢狀： ['links'] = [ { id: 1, owner: { id: 'u1', avatar: 'old' } }, { id: 2, owner: { id: 'u1', avatar: 'old' } }, // 重複！ ] Normalized： { users: { 'u1': { avatar: 'old' } }, // 只存一份 links: { '1': { owner_id: 'u1' }, '2': {...} } } // 改 user u1 的 avatar 一次，所有引用點自動更新
  • Apollo Client / RTK Query：內建 normalized cache
  • React Query / Vue Query：簡單 key-value cache（更輕量）

面試版回答

Multi-view 同步的核心是「single source of truth + publish-subscribe」——所有 view 訂閱同一份資料，資料變動時自動通知所有訂閱者。三層處理：

基礎機制：用 state library 或 query library 作為共享資料源，所有 view 透過 hook 訂閱。

更新策略：高風險用 pessimistic（等 server 確認），低風險用 optimistic（先更新 UI，失敗 rollback），即時應用可用 pending state。

Cache 同步管理：用 invalidateQueries 通知所有相關 query 重新 fetch。關鍵是要識別 cache 之間的依賴——例如改 profile 後，連帶顯示 profile 縮圖的 links cache 也要失效。

進階場景，當應用有大量關聯資料時，可以考慮 normalized cache（Apollo Client）——更新一次自動同步所有引用點。這是 React Query 跟 Apollo 的核心架構差異，也是工具選型的決策點。

3WebSocket 韌性設計

題目原文："Designing for network failures and WebSocket resilience — how do you handle reconnection, message queuing, and UI state when a real-time connection drops?"

連線會斷的真實原因（不是邊緣案例）

  • 網路切換（Wi-Fi 切 4G、走進電梯）
  • 背景分頁（手機切到背景自動斷）
  • 電腦休眠
  • Server 重啟
  • 負載平衡器超時（很多 LB 預設 60 秒沒活動就斷）
  • 防火牆 / Proxy 切斷長連線
🎯 核心精神

「承認連線必然會斷，把斷線當作正常情況處理」

子問題 1：Reconnection 策略

天真做法（會出大事）

ws.onclose = () => connect()  // ❌ 立刻重連

Thundering Herd Problem：1 萬個 client 同時重連 → 把 server 再次打掛。

正解：Exponential Backoff with Jitter

let retryDelay = 1000
const MAX_DELAY = 30000

ws.onclose = () => {
  const jitter = Math.random() * 1000
  setTimeout(connect, retryDelay + jitter)
  retryDelay = Math.min(retryDelay * 2, MAX_DELAY)
}

ws.onopen = () => { retryDelay = 1000 }

時間軸：1s → 2s → 4s → 8s → 16s → 30s → 30s...
加 jitter（隨機抖動）讓 1 萬個 client 重試時間自然錯開。

進階：判斷值不值得重連

ws.onclose = (event) => {
  if (event.code === 1000) return         // 正常關閉，不重連
  if (event.code === 1008) {              // 政策違反（被 ban）
    showError('連線被拒絕')
    return
  }
  if (event.code === 1011) return         // server internal error
  scheduleReconnect()                     // 其他原因才重連
}

子問題 2：Message Queuing（Outbox）

const outbox = []

function sendMessage(text) {
  const message = { id: uuid(), text, status: 'pending' }

  if (ws.readyState === WebSocket.OPEN) {
    ws.send(JSON.stringify(message))
  } else {
    outbox.push(message)  // 進佇列
  }
  addMessageToUI(message)  // UI 立刻顯示「傳送中」
}

ws.onopen = () => {
  // flush 佇列（按順序）
  while (outbox.length > 0) ws.send(JSON.stringify(outbox.shift()))
}

Idempotency（防重複）

每條訊息帶 client 端產生的 UUID。Server 檢查 ID，已收過就回 ack 不重複處理——避免使用者重發造成重複訊息。

子問題 3：UI 連線狀態的四個階段

狀態UI 設計
CONNECTING全頁 loading
CONNECTED綠點 icon
RECONNECTING頂部非阻塞 banner：「連線中斷，正在重連...」
DISCONNECTED更明顯 banner + [重試] 按鈕
⚠ UX 重點

不要用 modal 阻擋使用者。使用者可能還在打字、讀訊息——斷線不該打斷他們。

訊息狀態的視覺呈現

⏳ 傳送中 ← 在 outbox 還沒送 ↗️ 已送出 ← 送到 server，還沒 ack ✓ 已送達 ← server 確認 ✓✓ 已讀 ← 對方已讀 ❌ 失敗 ← 重試多次仍失敗

LINE / Messenger 訊息旁邊那些小符號——揭露底層同步狀態讓使用者知道每條訊息的處境。

跟 Server State 整合（進階）

ws.onmessage = (event) => {
  const data = JSON.parse(event.data)
  if (data.type === 'profile_updated') {
    queryClient.setQueryData(['profile', data.userId], data.profile)
    // 不用重新 fetch，server 已經告訴我們新值
  }
}

WebSocket 成為「server 主動觸發的 cache 更新管道」。

面試版回答

WebSocket 韌性的核心是「承認連線必然會斷」。三層設計：

Reconnection：exponential backoff with jitter，1s / 2s / 4s / 8s 上限 30s，加隨機抖動避免 thundering herd。根據 close code 判斷該不該重連。

Message Queuing：client 端 outbox，斷線時訊息進佇列、UI 標示「傳送中」；連線恢復後 flush。每條訊息帶 idempotency key 避免重複處理。

UI State：四個階段，reconnecting 用非阻塞 banner，不打斷使用者操作。每條訊息有自己的狀態（pending / sent / delivered / failed），用 icon 揭露給使用者。

跟 server state 架構整合：WebSocket 推送可以透過 setQueryData 直接寫進 cache，不用重新 fetch。

4Folder 結構 + Code Splitting

兩種主流組織方式

By Type（按類型）

src/ ├── components/ (所有元件擠在這) ├── hooks/ ├── utils/ ├── api/ └── pages/
  • ✅ 新手友善（直覺）
  • ❌ 大專案重構難（一個功能的檔案散在四處）

By Feature（按功能）

src/ ├── features/ │ ├── auth/ ← 自包含 │ │ ├── components/ │ │ ├── hooks/ │ │ └── api/ │ ├── profile/ │ └── links/ ├── shared/ ← 跨 feature 共用 └── pages/
  • ✅ 結構反映業務邏輯
  • ✅ 重構容易（刪一個 feature 就刪一個資料夾）
  • ✅ 邊界清晰

實務做法：混合

「By Feature 為骨架，By Type 為細節」

src/ ├── features/ # 業務邏輯 ├── shared/ # 跨 feature 共用 ├── lib/ # 第三方服務封裝（supabase、analytics） ├── pages/ # 路由層 └── app/ # 全域設定（router、providers）

判斷東西該放哪：「這個被多少 feature 用？」

  • 1 個 → 進該 feature
  • 2-3 個 → 暫時放最常用的 feature
  • 4+ → 升級到 shared/

Rule of Three（重要原則）

💡 Rule of Three

第三次重複使用時才抽出共用。前兩次接受複製，避免過早抽象。

反模式：「我覺得這 button 將來會多處用，先放 shared」→ 半年後只一個地方用，shared 變垃圾場。

Separation of Concerns（檔案層級）

反模式：所有東西塞一個元件

function ProfilePage() {
  // state 管理 + API 呼叫 + 業務邏輯 + UI 渲染 ← 身兼四職
}

正解：分離

features/profile/ ├── api/profileApi.ts # 純 fetch ├── hooks/useProfile.ts # 資料 + state ├── hooks/useProfileStats.ts # 業務邏輯 ├── components/ │ ├── ProfileHeader.tsx # 純 UI │ └── ProfileStats.tsx # 純 UI └── pages/ProfilePage.tsx # 組合層

每個檔案只做一件事——改 API 不影響 UI、改 UI 不影響業務邏輯。

Code Splitting（三層切分）

1. Route-level（效益最大）

const StatsPage = lazy(() => import('./pages/StatsPage'))

<Suspense fallback={<Spinner />}>
  <Route path="/stats" element={<StatsPage />} />
</Suspense>

Next.js / Nuxt 自動做。

2. Component-level

const YouTubeEditor = lazy(() => import('./YouTubeEditor'))
// 只在使用者選 YouTube 類型才下載

Myverse 適用：把不同卡片類型拆成獨立元件 + dynamic import。

3. Library-level

const Chart = lazy(() => import('chart.js'))         // 重型 library
const MarkdownEditor = lazy(() => import('@uiw/react-md-editor'))

Code Splitting 的成本與限制

❌ 反模式

把每個小元件都 lazy load——每個小檔案是一個 HTTP request → 瀑布式載入 → 反而變慢。

判斷標準：splitting 對象至少 50KB 以上才有意義。

進階技巧：Preloading

<Link
  to="/stats"
  onMouseEnter={() => import('./pages/StatsPage')}  // hover 時預載
>

Next.js 的 <Link> 自動做——連結進入 viewport 時開始下載。

面試版回答

By Feature 為骨架、By Type 為細節。每個 feature 自包含（components / hooks / api），跨 feature 的共用放 shared/，但堅持 Rule of Three——前兩次重複接受複製，第三次才抽。

這比 By Type 強的地方在「程式碼結構反映業務結構」——刪一個功能就刪一個資料夾，新人看 features 就知道專案做什麼。

Code splitting 我分三層：route level 用 lazy + Suspense（效益最大）、component level 切重型元件、library level 切重型第三方 library。但小於 50KB 不切——每個 chunk 是一個 HTTP request，過度切分反而慢。

By Feature 跟 code splitting 是相輔相成的——每個 feature 天然是一個 splitting 單位。

5React 渲染機制

Re-render 不等於 UI 更新

⚠ 我曾誤答

「UI 更新時就會 re-render」——因果反了。

正確的因果鏈

觸發條件發生 ↓ React 決定元件要 re-render ↓ 元件 function 重新執行（這就是 re-render） ↓ 產生新的 Virtual DOM ↓ React 比對新舊 Virtual DOM（reconciliation） ↓ 有差異 → 更新真實 DOM（這才是 UI 更新）
🔑 關鍵

Re-render 100 次，如果 Virtual DOM 都一樣，真實 DOM 一次都不會變。但 JS 重新執行 + Virtual DOM 計算 + 比對的 CPU 成本還是發生了。

效能優化的目標 = 減少不必要的 re-render。

觸發 re-render 的三個原因

  1. 元件自己的 state 改變（且 Object.is 比對下新舊值不同）
  2. 元件的 props 改變
  3. 元件訂閱的 Context 值改變
⚠ 我曾誤答

「useState 的 setter 被呼叫一定會 re-render」——錯。

const [count, setCount] = useState(0)
setCount(0)  // 不會 re-render（state bailout）

const [user, setUser] = useState({ name: 'Bo' })
setUser({ name: 'Bo' })  // ❗ 會 re-render（新物件，新 reference）

父 re-render，子預設一定跟著 re-render

預設情況下，父元件 re-render，所有子元件都 re-render——不管 props 有沒有變。

function Parent() {
  const [count, setCount] = useState(0)
  return (
    <div>
      <button onClick={() => setCount(c => c + 1)}>+1</button>
      <Child name="Bo" />  {/* count 變動時，Child 也會 re-render */}
    </div>
  )
}

讓子元件「props 沒變就不 re-render」的兩個條件

  1. ✅ 子元件被 React.memo 包起來
  2. ✅ 傳給子的 props 在淺比較下沒改變
⚠ 我曾誤答

以為跟「子元件的 state」有關——其實完全無關。

6Reference（引用）—— React 效能的核心前提

JavaScript 的兩種比較規則

Primitive（number、string、boolean、null、undefined）：比值

5 === 5         // true
'abc' === 'abc' // true

Object 型別（object、array、function）：比 reference

const a = { name: 'Bo' }
const b = { name: 'Bo' }
a === b   // false ← 看起來一樣，但是不同物件

const c = a
a === c   // true ← 同一個 reference

函式也是物件

const fn1 = () => console.log('hi')
const fn2 = () => console.log('hi')
fn1 === fn2  // false

串連到 React：為什麼這個觀念是核心

每次元件 re-render，function 重新執行，裡面定義的物件、函式、陣列都是新建立的 reference：

function TodoList({ todos }) {
  const handleDelete = (id) => { ... }  // 每次 render 都是新函式 reference
}

React.memo 做淺比較時，發現 reference 變了 → 認為 props 變了 → 子元件還是 re-render。

7React.memo / useCallback / useMemo

React.memo：在子元件加檢查點

作用：淺比較 props 後決定要不要 re-render。

const Child = React.memo(function Child({ name }) {
  return <div>{name}</div>
})
⚠ 我曾誤答

「React.memo 讓沒有用到 props 的 UI 不會跟著 re-render」

正確：「React.memo 讓元件在 props 沒有改變時跳過 re-render」（差別：元件可能有 props，只是這次沒變）

useCallback：記住函式 reference

const handleClick = useCallback(() => {
  console.log('clicked')
}, [])  // 空依賴陣列 → reference 永遠不變

useMemo：記住值 reference 或計算結果

作用 1：穩定物件 / 陣列 reference

const config = useMemo(() => ({ theme: 'dark' }), [])

作用 2：避免昂貴計算（這是 useCallback 沒有的用途）

const stats = useMemo(() => {
  return rawData.reduce((acc, item) => computeHeavyStats(acc, item), initial)
}, [rawData])

兩者的差別

工具穩定 reference避免重算
useCallback✅ 函式❌（建立函式幾乎免費）
useMemo✅ 物件 / 陣列 / 計算結果✅ 昂貴計算

判斷標準

等號左邊的變數最後拿到的是什麼？

  • 拿到函式 → useCallback
  • 拿到值（物件、陣列、字串、數字、計算結果） → useMemo
⚠ 我曾誤答

把 const title = useMemo(() => 'User Dashboard', [...]) 寫成 useCallback——被「箭頭函式」誤導。

第二個錯：簡單字串拼接根本不用 useMemo——字串是 primitive，不需要穩定 reference。

何時該用 useCallback / useMemo（重要！）

⚠ 反模式警告

很多新手到處包，這是反優化。它們本身有記憶 + 比較成本。

只有以下三種情境才該用

  1. 下游元件用了 React.memo，需要穩定 reference 才能讓 memo 生效
  2. 這個值是其他 hook 的依賴（例如 useEffect 的 dependency array）
  3. 計算成本真的很高（只用於 useMemo）

判斷流程

這個東西會被當作 prop 傳給用了 React.memo 的子元件嗎？ → 是 → 可能需要包 這個東西會被放在其他 hook 的依賴陣列裡嗎？ → 是 → 可能需要包 這個計算本身是不是很慢（毫秒等級以上）？ → 是 → useMemo 包 三個都不是 → 不要包，直接寫

三個工具是組合拳

React.memo（在子元件） ↓ 需要搭配 useCallback / useMemo（在父元件，讓傳下去的 props reference 穩定）
  • 只用 React.memo 不用 useCallback / useMemo → memo 失效（TodoList 經典陷阱）
  • 只用 useCallback / useMemo 不用 React.memo → 完全沒意義

經典陷阱：TodoList memo 失效

function TodoList({ todos }) {
  const handleDelete = (id) => { ... }  // ❌ 每次 render 新函式
  return todos.map(todo =>
    <TodoItem key={todo.id} todo={todo} onDelete={handleDelete} />
  )
}

const TodoItem = React.memo(function TodoItem({ todo, onDelete }) { ... })

問題：TodoItem 雖然 memo，但 onDelete reference 每次都變 → memo 失效，全部 re-render。

修法 1：handleDelete 用 useCallback 包

修法 2（容易忘）：更新 todos 陣列時，保留沒變的 todo 的 reference

// ✅ 正確：只替換改動的那個
setTodos(todos.map(t => t.id === 1 ? { ...t, text: 'new' } : t))

// ❌ 錯誤：每個 todo 都建立新物件
setTodos(todos.map(t => ({ ...t })))

這就是「state 要 immutable」更深的原因——保持 reference 穩定讓下游 memo 生效。

面試版回答

useCallback 跟 useMemo 不是免費的——它們本身有記帳和比較的成本。所以不是「隨時可以加來優化」，而是要有具體的目的：

1. 配合下游的 React.memo 使用，讓 props 的 reference 穩定
2. 避免在 useEffect 等 hook 的依賴陣列裡造成不必要的觸發
3. 包住真正昂貴的計算（這只有 useMemo 適用）

沒有這三種情境的話，包這兩個 hook 反而是反優化。我看過很多專案到處包 useCallback，結果效能反而比沒包還差，就是這個原因。

8Async Flow + Custom Hook

「天真寫法」會踩到的三個 bug

useEffect(() => {
  setLoading(true)
  fetch(`/api/users/${userId}`)
    .then(res => res.json())
    .then(setUser)
}, [userId])

Bug 1：Race Condition

  • 使用者快速切換 user 1 → user 2
  • request 1 比 request 2 晚回來 → 蓋掉 user 2 的資料

Bug 2：Memory Leak

  • 元件 unmount 後，fetch 還在路上
  • 回應後執行 setState → React 警告「Can't perform state update on unmounted component」

Bug 3：UI 閃爍

  • userId 1 → 2 → 3 快速變動
  • request 1 先回來：loading 變 false，顯示 user 1
  • 後面 request 陸續回來：畫面在閃

根源：fetch 一旦發出就「失控」，無法取消。

解法：AbortController + cleanup

useEffect(() => {
  const controller = new AbortController()

  setLoading(true)
  fetch(url, { signal: controller.signal })
    .then(res => res.json())
    .then(data => {
      setUser(data)
      setLoading(false)
    })
    .catch(err => {
      if (err.name === 'AbortError') return  // 不當錯誤處理
      setError(err)
    })

  return () => controller.abort()  // ✨ cleanup
}, [userId])

Cleanup 執行的兩個時機（重要！）

⚠ 我曾漏答一個時機

只說「依賴值有變動時」

完整的兩個時機：

  1. 下次 effect 即將執行前（依賴變動 → 先 cleanup → 再執行新 effect）
  2. 元件 unmount 時（最後一次 cleanup）

同一段 cleanup 程式碼，同時解決 race condition（時機 1）和 memory leak（時機 2）——這就是 useEffect 設計的精巧之處。

Custom Hook 的四大好處

⚠ 我曾答得太簡略

「好維護」

完整四個原因：

  1. 邏輯重用（DRY）：多個元件要 fetch 不用每個都寫一遍
  2. 關注點分離：元件只關心 UI，fetch / race condition / cleanup 細節藏在 hook 裡
  3. 可測試性：用 renderHook 獨立測試
  4. 集中升級邏輯入口：未來要加 retry、cache、token 都改一個檔案全站升級 ← 最重要的價值

9Debounce / Throttle / useRef

核心差別

特性DebounceThrottle
觸發時機等事件停止後一段時間固定間隔執行一次
最後一次觸發保證會執行不一定執行
適合場景結果取最後一次過程中要回應
比喻電梯等人關門電梯定時開

判斷標準

🔑 一句話記憶

「結果取最後一次 → Debounce；過程中要回應 → Throttle」

⚠ 我曾誤判

把「動態 / 即時更新」當成 debounce 的關鍵字。

正確：「動態 / 即時 / 過程中持續顯示」 = throttle。「停下來才 / 等使用者完成」 = debounce。

場景對照

Debounce 場景

  • 搜尋框 autocomplete
  • 表單 input 即時驗證
  • 自動儲存草稿
  • 視窗 resize 後重新計算 layout

Throttle 場景

  • Scroll 事件監聽（無限滾動觸發）
  • Mouse move 追蹤
  • 防止連點按鈕（API rate limiting）
  • 拖拉地圖時即時顯示座標 ← 我曾誤答 debounce
  • 字數即時顯示 ← 但其實連 throttle 都不需要（純前端零成本）
⚠ 重要的進階檢查

Debounce / throttle 是用來節流昂貴操作：API request、複雜計算、大量 DOM 更新。

對 text.length 這種便宜到不能再便宜的操作，每次 onChange 都直接更新 UI 是 OK 的。硬包 throttle 反而是反優化。

React 裡實作 debounce：useRef

錯誤做法：直接定義在元件裡

function SearchBar() {
  // ❌ 每次 re-render 都建立新 debounced 函式，計時器 timerId 也是新的
  const debouncedSearch = debounce((kw) => {...}, 500)
}

正確做法：useRef

const debouncedSearchRef = useRef(
  debounce((kw) => fetch(`/api/search?q=${kw}`), 500)
)
// 使用：debouncedSearchRef.current(kw)

Leading vs Trailing Edge（加分題）

  • Trailing edge debounce（預設）：「停下來才執行」——適合搜尋
  • Leading edge debounce：「第一次立刻執行，之後一段時間忽略」——適合防止重複提交

useRef 的真面目

⚠ 常見誤解

「useRef 只能綁 DOM 元素」

真正的本質：跨 re-render 持久存在、修改不會觸發 re-render 的容器。

Use Case 1：綁定 DOM

const inputRef = useRef(null)
useEffect(() => inputRef.current.focus(), [])
return <input ref={inputRef} />

Use Case 2：存任何「不該觸發 re-render 的值」

  • 計時器 ID
  • 上一次的 props 值
  • debounced 函式
  • 內部計數器

useRef vs useState

特性useStateuseRef
修改後觸發 re-render✅❌
適用需要顯示在 UI 的資料純內部資料

判斷：這個值改變時，畫面要不要更新？要 → useState；不要 → useRef。

10Virtualization（虛擬化）

解決的問題

大列表（10000 筆訊息）直接渲染 → 建立 10000 個 DOM → 慢、記憶體爆炸、滾動卡頓。但使用者一次只看到 10-20 筆。剩下 9980 筆是浪費。

核心思想

「只渲染使用者實際看得到的，其他不要建立 DOM」

實作要點

  • 用「假高度容器」騙瀏覽器，讓滾動條長度看起來正常
  • 用絕對定位把實際渲染的元素放在正確位置
  • 滾動時動態替換哪些元素該渲染

工具

  • react-window：輕量、簡單，固定高度容易
  • @tanstack/react-virtual：現代、支援動態高度

限制與權衡（面試考點）

❗ Virtualization 不是免費的
  1. Ctrl+F 搜尋失效：瀏覽器搜尋只能搜 DOM 裡的元素
  2. SEO 不友善：爬蟲只看 DOM
  3. Accessibility 要額外處理：螢幕閱讀器可能會迷路
  4. 動態高度很難：要先測量，會 layout shift

我曾誤判：450 萬筆紀錄該不該 virtualize？

錯題復盤

我答：「不用，數量不影響效能」

正確判斷：

資料量策略
< 50 筆直接渲染
50-200直接渲染 + React.memo
200-2000應該 virtualize
2000-10000必須 virtualize + 思考分頁
> 10000必須 virtualize + 必須分頁 / 篩選

更深的層次：質疑前提

💡 senior 思維

對 450 萬筆瀏覽紀錄，最厲害的設計是讓使用者根本不需要面對大資料。

Step 1: 使用者真的需要看到「所有」紀錄嗎？ → 不需要，用聚合 / 統計 / 圖表呈現（資料量降到幾十個數字） Step 2: 能不能用篩選 / 分頁縮小資料量？ → server-side pagination + 普通列表（每頁 50 筆） Step 3: 真的要快速瀏覽全部時才用 virtualization

核心觀念：UI 應該回答使用者的問題，不是把資料庫倒出來。資料 ≠ 資訊。450 萬筆紀錄是「資料」，「你的母親節活動瀏覽量比上週高 230%」是「資訊」。

綜合應用：高頻更新大列表

「每秒 100 筆新訂單，列表已 5000 筆」這類場景需要 5 個工具一起用：

  1. Virtualization：DOM 只保留 20 個元素
  2. React.memo：訂單沒變就不 re-render
  3. useCallback / useMemo：保持 props reference 穩定
  4. throttle：每秒最多更新 10 次，不是每筆都 setState
  5. immutable update：保留沒變的訂單 reference，讓 memo 跳過

  • 形式：一句話 AI 提示
  • 互動：旁邊有展開鍵看詳情

對應的設計原則（面試金句）

設計選擇對應原則
卡片下方Proximity——建議跟對應物件放一起

核心原則：Ambient over Proactive

  • Proactive：AI 主動跳出，搶使用者注意力 ❌
  • Ambient：AI 在那裡但不搶戲，使用者用自己節奏決定看不看 ✅

排序的依據（面試官會追問）

  • 預期影響力（預估點擊率提升幅度）
  • 執行成本（使用者要花多少時間）
  • 時效性（限時優惠快過期 > 一般排版建議）
  • 使用者過去的行為（他常採納哪類建議）

我曾答：每次編輯都跳

校正：會變成狼來了，使用者會無視所有提示。需要降低觸發頻率、限制同時數量、忽略後變淡等機制。

12AI Agent — Interaction Model

題目原文："How would you design the interaction model? (e.g., proactive suggestions vs. on-demand triggers, inline vs. side panel)"

雙模式架構

模式適合情境形式
Inline 建議
（卡片下方一句話）		微調個別元素被動、ambient
對話式 agent
（side panel）		有明確目標、需多步驟主動、goal-oriented

對應使用者的兩種心智狀態：

  • Tweaking mode（隨便編輯一下）→ inline
  • Task-driven mode（我要做母親節活動）→ side panel

對話式 agent 三個位置選擇

選項優點缺點
浮動按鈕 + 彈出對話框不佔版面對話時看不到後台
Side panel
（從右側滑出）		可邊對話邊看版面變化 ✅佔螢幕空間
全螢幕對話對話體驗最好完全脫離後台情境

為什麼選 Side Panel：See-while-you-do

使用者跟 AI 對話的同時，可以即時看到後台版面變化。對「AI 幫我動手做」特別重要——使用者需要驗證 AI 做的事符不符合預期。

進階觀念：Trust Calibration（面試金句）

使用者不會一開始就完全信任 AI。需要：

  1. 看著 AI 做事
  2. 確認結果合理
  3. 慢慢放手讓 AI 做更多

Side panel 設計剛好支援這個漸進信任過程。

面試版回答

我會設計成雙模式：

Inline ambient 建議——卡片下方一句話 + 展開鍵，最多顯示 3 條。適合使用者在微調時，AI 提供被動的針對性建議。

Side panel 對話式 agent——從右側滑出的固定面板，使用者用自然語言描述目標，AI 動手建立卡片、調整排版，使用者在左邊即時看到變化。適合有明確目標、需多步驟的任務。

兩個模式對應 tweaking mode 和 task-driven mode。Side panel 的設計也支援 trust calibration——使用者邊對話邊驗證 AI 的動作，建立漸進的信任。

13AI Agent — Data + Pipeline

題目原文："What data would the agent need to 'see' from the UI state, and how would you architect that pipeline?"

五層資料

⚠ 我曾漏列 2 層

原本只想到 UI State、Analytics、Audience，漏了 Creator Identity 和 User Intent Signals。

資料層內容在回答什麼
UI State目前頁面卡片、版型、設定「現在長什麼樣？」
Analytics Data過去 30/60/90 天的點擊、瀏覽、轉換「過去表現如何？」
Audience Context受眾人口統計、興趣、訪問時段「給誰看的？」
Creator Identity創作者領域、風格、品牌調性「這個創作者是誰？」
User Intent Signals當下行為、對話歷史、近期操作「使用者現在想幹嘛？」

為什麼 Creator Identity 重要

舉例

同樣一句「幫我做母親節活動」，AI 要根據創作者的領域給完全不同的建議——美妝部落客建議口紅，親子部落客建議家庭活動，美食創作者建議餐廳。

為什麼 User Intent Signals 重要

「使用者沒有直接說，但 AI 應該感知到」的訊號：

  • 停在哪張卡片很久（暗示煩惱）
  • 最近一次編輯什麼（暗示方向）
  • 剛剛刪掉什麼（暗示不喜歡）
  • 對話歷史說過什麼（「我不喜歡紅色」）

靜態 vs 動態資料

資料性質傳送策略
Creator Identity靜態Session 開始送一次，放 system prompt
Audience Context靜態Session 開始送一次
UI State動態每輪都送最新的
Analytics Data看情況不是每輪送全部，要按需查詢 ← 重點
User Intent Signals動態每輪都送（對話歷史累積）

工程術語：static context vs dynamic context，或 system prompt vs runtime context。

Tool Use 模式（核心架構）

問題：每次都把 5 層全部塞進 prompt 會 token 爆炸。

傳統做法（被動接收）

使用者問題 → 前端塞所有資料 → AI 回應

Tool Use 做法（主動查詢）

使用者問題 → 前端只送基本 context ↓ AI：「我需要知道過去節日活動的表現」→ 呼叫 get_analytics(filter: "holiday") ↓ 前端執行查詢 → 回傳結果 ↓ AI 基於 tool 結果產生最終回應

AI 變成主動去拿它需要的資料。Anthropic 叫 tool use，OpenAI 叫 function calling。

Myverse Pipeline 完整架構

[Session 開始] ↓ Frontend 收集 static context: - Creator Identity（從 Supabase user profile） - Audience Context（從 analytics 摘要） ↓ 塞進 system prompt → 初始化對話 [每輪對話] ↓ 使用者輸入訊息 ↓ Frontend 加上動態 context： - 當下 UI State（Pinia / Vue state） - 對話歷史 ↓ 送給 AI ↓ AI 判斷：需要更多資料嗎？ ├─ 不需要 → 直接回應 └─ 需要 → 呼叫 tool（get_card_analytics） ↓ Frontend 執行 → 回傳 ↓ AI 基於 tool 結果產生回應 ↓ 回應顯示在 side panel ↓ 若 AI 要修改卡片 → 呼叫 update_card / reorder_cards tool → 前端執行 → UI 即時更新

面試版回答

Pipeline 我會分三層處理：

靜態層（creator identity、audience）在 session 開始時收集一次，放進 system prompt。

動態層（UI state、對話歷史）每輪對話都更新，附加在 user message。

按需查詢層（analytics 等大量資料）我不會預先塞，而是用 tool use 模式讓 AI 主動呼叫前端的查詢工具。避免 token 爆炸，也讓 AI 能精準取得需要的資料。

同樣地，AI 修改 UI 也是透過 tool use——呼叫 update_card、reorder_cards 這類前端工具，前端執行後 UI 即時更新，使用者邊對話邊看到後台變化。

14AI Agent — Latency Handling

題目原文："How would you handle AI response latency in a real-time UI — skeleton states, streaming text, confidence indicators?"

AI 延遲的三個痛點

⚠ 我原本只想到 1 個解法（輸入中圖示）

實際上要四層。

痛點使用者 OS解法
不知道系統有沒有在動「是不是當機了？」Loading indicator
等待時間感覺很長「怎麼這麼慢...」Streaming
不知道會給什麼答案「等了半天會白等？」Process transparency
不知道答案是否可信「這 AI 講的我可以信嗎？」Confidence indicator

第 1 層：Loading Indicator

最基本——「輸入中」圖示，告訴使用者系統有在動。

第 2 層：Streaming（核心技巧）

ChatGPT 之所以「感覺很快」的秘密。

  • 沒 streaming：等 8 秒 → 一次看到完整回應
  • 有 streaming：等 0.5 秒 → 文字一個字一個字跳出 → 8 秒後跳完

總時間一樣，體感差很多。為什麼？

  1. Time to First Token（TTFT）變短——0.5 秒看到動靜
  2. 使用者邊讀邊等
  3. 方向錯了可以提早打斷

技術：Anthropic / OpenAI API 都支援 streaming（Server-Sent Events）。

第 3 層：Process Transparency（進階技巧）

不只佔位，預告 AI 正在做什麼：

🤔 正在分析你的受眾資料... ⏳ 正在查詢過去節日活動的表現... ✏️ 正在生成卡片建議...

對應到 AI 內部呼叫的 tool use 步驟。每呼叫一個 tool 就更新一行狀態。

價值

  1. 使用者知道 AI 不是在轉圈圈，是真的在做事
  2. 預期到 AI 會給什麼方向的答案
  3. 某步特別久時也能理解原因

第 4 層：Confidence Indicator

AI 給的建議不是每條都一樣可靠：

✓ 數據顯示（高信心）：你的限時優惠卡片過去點擊率最高 💡 建議（中信心）：可以新增一張母親節主題卡片 🔮 推測（低信心）：粉紅色可能更符合節日氛圍

回到 trust calibration——讓使用者學會在哪裡信任 AI、在哪裡保留自己的判斷。

完整時間軸

0 秒：使用者送出 0.1 秒：「輸入中」圖示 0.5 秒：Process transparency「🤔 正在分析受眾資料...」 2 秒：tool 完成，更新「⏳ 正在查詢節日活動表現...」 4 秒：開始 streaming 文字 8 秒：完整回應，每條建議標 confidence indicator

面試版回答

AI 延遲不是用單一策略解決，我會用四層方法組合：

第一層 Loading indicator：基本「輸入中」圖示，告訴使用者系統有在動。

第二層 Streaming：用 SSE 串流逐字顯示，把 TTFT 降到 1 秒內。使用者可以邊讀邊等，必要時提早中斷重問。

第三層 Process transparency：因為我們用 tool use 模式，可以把每個 tool 呼叫的狀態顯示給使用者——「正在查詢數據」、「正在生成建議」。讓等待時間從黑盒子變成可見的工作流程。

第四層 Confidence indicator：每條建議標示信心等級——數據支撐、主觀建議、推測——用不同視覺標示。幫助使用者做 trust calibration。

整體目標是讓 AI 從「慢」變成「可預期且透明」——使用者不會因為等 8 秒就覺得卡，反而會覺得 AI 在認真做事。
